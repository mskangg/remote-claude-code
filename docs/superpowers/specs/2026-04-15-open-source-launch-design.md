# Remote Claude Code 오픈소스 런치 디자인

## 문서 목적

이 문서는 Remote Claude Code를 public 오픈소스로 공개할 때, GitHub 방문자가 빠르게 제품 가치를 이해하고 스타를 누르게 만드는 제품 페이지/README/설치 경험의 방향을 정리한다.

이번 단계의 우선순위는 다음과 같다.

1. GitHub 스타를 얻을 수 있는 첫인상 만들기
2. 실제 사용자가 빠르게 설치·검증·첫 실행까지 갈 수 있게 만들기

핵심 타겟은 팀 전체가 아니라 **개인 해커 / 인디 개발자**다.

---

## 문제 정의

현재 저장소에는 제품의 핵심 능력이 이미 존재한다.

- Slack에서 `/cc`로 세션 진입
- tmux 위에서 기존 Claude Code 세션 실행/유지
- Slack thread에서 명령 전달
- 진행 상태와 최종 응답 relay
- `doctor`를 통한 설치/환경 검증
- 데모 자산 초안(`docs/hero-mock-v18.html`)

하지만 public 오픈소스 관점에서는 “무엇을 만들고 있는가”보다 **“왜 지금 당장 별을 눌러야 하는가”**가 더 중요하다.

즉 현재 부족한 것은 핵심 기능 자체보다:

- 한 줄 포지셔닝
- README 첫 화면의 설득력
- 설치 경험의 심리적 마찰 최소화
- `doctor`를 통한 신뢰 확보
- 데모와 카피의 일관된 서사

---

## 핵심 제품 포지셔닝

### 한 줄 제품 약속

> 언제 어디서든, 휴대폰으로도 내가 이미 작업 중인 Claude Code 환경에 그대로 일을 시킬 수 있다.

### 확장 설명

- 새 에이전트를 배우는 제품이 아니다.
- 새 원격 개발환경으로 옮겨 타는 제품이 아니다.
- 지금 일하던 로컬 PC 또는 클라우드 작업환경의 Claude Code를 그대로 이어 쓰는 제품이다.
- Slack은 새로운 실행 엔진이 아니라 **원격 UI**다.

### 사용자 관점의 가치

사용자는 새로운 시스템에 적응하지 않는다. 그냥:

- 평소처럼 Claude Code로 일하다가
- 자리를 비우거나
- 휴대폰만 들고 있거나
- Slack에 들어가서
- 같은 세션에 일을 계속 시킨다.

이 continuity가 이 제품의 가장 강한 차별점이다.

## 타겟 사용자

### 1차 타겟

**개인 해커 / 인디 개발자 / Claude Code 헤비 유저**

이들은 다음 조건에 강하게 반응한다.

- 이미 Claude Code를 자주 쓴다
- 작업환경을 바꾸고 싶지 않다
- 이동 중에도 작업 continuity를 원한다
- 설치 과정이 길면 이탈한다
- 하지만 AI가 대신 설치해주는 흐름은 자연스럽게 받아들인다

### 제외할 포지셔닝

초기 런치에서는 다음을 전면에 두지 않는다.

- 팀 협업 플랫폼
- 엔터프라이즈 오케스트레이션
- transport-neutral architecture
- 복잡한 시스템 설계 설명

이런 메시지는 제품의 내부 진실일 수는 있어도, 첫 공개에서 스타를 부르는 메시지는 아니다.

## 설치 경험 디자인

### 제품 원칙

설치는 반드시 다음 중 하나여야 한다.

1. 진짜 딸깍 설치
2. 또는 Claude Code가 대신 설치해줄 수 있을 만큼 단순한 설치

즉 “손으로 긴 문서를 읽고 수동으로 조립하는 설치”는 타겟 사용자에게 맞지 않는다.

### 왜 이 기준이 맞는가

타겟 사용자는 Claude Code 사용자다. 따라서:

- 완전 자동 설치가 가장 좋고
- 그게 어려우면 “이 repo 설치해줘”라고 Claude Code에게 맡길 수 있으면 된다

이건 설치 UX를 낮추는 게 아니라, 사용자 현실에 맞게 재정의하는 것이다.

### 설치 파트에서 보여줘야 할 것

- prerequisites 최소화
- 복붙 가능한 명령
- AI가 실행하기 쉬운 단계 순서
- 설치 후 바로 `doctor`
- 그 다음 앱 실행
- 그 다음 `/cc`

## Doctor 위치와 역할

`doctor`는 보조 명령이 아니라 **신뢰 형성 장치**다.

### README에서의 역할

- 설치가 제대로 되었는지 즉시 검증
- 사용자가 실패 원인을 구조적으로 파악
- “이 프로젝트는 운영 관점도 생각했다”는 신뢰 제공

### README에서의 배치

`doctor`는 README 뒤쪽에 숨기지 않는다.

권장 흐름:
1. install
2. config
3. `doctor`
4. run app
5. Slack에서 `/cc`

이 흐름은 “설치 → 검증 → 첫 성공”의 심리와 맞다.

## 대표 사용 사례

README와 launch post에 반복적으로 사용할 대표 시나리오는 3개로 제한한다.

### 1. 외출 중 휴대폰으로 작업 이어가기
자리를 비운 상태에서 휴대폰으로 Slack에 들어가 현재 Claude Code 세션에 일을 이어서 시킨다.

### 2. 이동 중 코드 리뷰 / 파일 검토 지시
간단한 분석, 파일 검토, 다음 액션 정리 같은 일을 Slack thread로 전달한다.

### 3. 장기 작업 흐름 유지
긴 작업을 하나의 세션/스레드 단위로 이어가며 상태와 응답을 추적한다.

이 사용 사례들은 모두 “새 워크플로우”가 아니라 “기존 워크플로우의 연장”으로 설명해야 한다.

## How it works 섹션 방향

이 섹션은 README 중반 이후에 배치한다.

설명은 짧고 명확해야 한다.

### 설명해야 할 핵심

- Slack은 원격 UI다
- Claude Code는 로컬 또는 클라우드의 실제 작업환경에서 돈다
- tmux/session/hook 기반으로 상태와 응답을 relay한다
- 사용자는 작업환경을 옮기지 않는다

### 설명 방식

구조적 진실은 유지하되, 내부 아키텍처 자랑처럼 보이지 않게 한다.

즉 “왜 믿을 수 있는가”를 설명하지 “우리 구조가 얼마나 멋진가”를 설명하지 않는다.

## 최종 권장 방향

이 프로젝트는 초기 공개에서 다음 한 문장으로 기억되어야 한다.

> Slack으로, 심지어 휴대폰으로도, 내가 이미 작업 중인 Claude Code 환경에 그대로 일을 시킬 수 있는 오픈소스 도구.

그리고 그 다음 인상은 다음이어야 한다.

- 설치는 무겁지 않다
- Claude Code가 대신 설치해줄 수도 있다
- `doctor`로 검증 가능하다
- 실제로 `/cc`까지 빠르게 간다

즉 이 제품의 첫 공개 전략은 **기능 설명 중심**이 아니라,
**작업 continuity + 낮은 설치 마찰 + 검증 가능성** 중심으로 설계한다.
